# Регламент: Авторизація та система доступів до внутрішнього порталу

> [!WARNING]
> Цей документ має статус `draft`. Не є офіційним до підтвердження редактором.

## Мета

Встановити єдині правила авторизації та управління доступами до внутрішнього порталу компанії, щоб:

- закрити службові документи від сторонніх;
- розмежувати доступ за ролями;
- відокремити публічний сайт від внутрішньої бази знань;
- зменшити ризик витоку документів, помилкового доступу та хаотичного розповсюдження матеріалів.

## Область застосування

Регламент застосовується до:

- внутрішнього порталу;
- корпоративної wiki;
- захищених розділів сайту;
- регламентів;
- посадових інструкцій;
- управлінських документів;
- вкладень, файлів і службових матеріалів.

## Базовий принцип архітектури

> [!IMPORTANT]
> Публічний сайт і внутрішня база знань не повинні жити в одному рівні доступу.

Рекомендована модель:

- `public` — публічний сайт для клієнтів;
- `internal` — окремий внутрішній портал / wiki з авторизацією;
- `admin` — окремий рівень керування доступами та контентом.

### Рекомендований формат розміщення

| Рівень | Призначення | Приклад |
|---|---|---|
| Публічний | Контент для клієнтів | `secretshop.ua` |
| Внутрішній | Регламенти, інструкції, база знань | `wiki.secretshop.ua` або `staff.secretshop.ua` |
| Адміністративний | Керування користувачами, ролями, чернетками | окремий admin-рівень |

## Модель доступу

Базова модель доступу будується за принципом:

- **role-based access control** — доступ визначається роллю;
- **least privilege** — кожен співробітник бачить лише те, що потрібно для його ролі;
- **deny by default** — новий користувач не отримує зайвий доступ автоматично.

## Ролі

| Роль | Призначення |
|---|---|
| `admin` | Повний доступ до системи, користувачів, ролей, налаштувань і контенту |
| `hq` | Центральний офіс / управлінська команда |
| `director` | Директор магазину |
| `supervisor` | КЗ / адміністратор / керівник зміни |
| `seller` | Продавець |
| `guest` | Неавторизований або користувач без наданого доступу |

## Рівні доступу до контенту

| Рівень доступу | Що означає |
|---|---|
| `view` | Перегляд документа |
| `download` | Завантаження вкладень / файлів |
| `comment` | Коментування / фідбек |
| `edit` | Редагування контенту |
| `publish` | Публікація / переведення зі статусу draft |
| `admin` | Керування ролями, політиками та доступами |

## Окремий блок для директорів

Для ролі `director` рекомендується виділити окремий захищений розділ:

- посадова інструкція директора;
- регламенти управління магазином;
- правила контролю KPI;
- ескалаційні сценарії;
- кадрові та організаційні SOP;
- регламенти конфліктних ситуацій.

> [!NOTE]
> Матеріали для директорів не повинні бути видимі продавцям і не повинні індексуватись публічними пошуковими системами.

## Правила авторизації

1. Кожен співробітник повинен мати персональний обліковий запис.
2. Спільні логіни типу `director01` або `storeadmin` не рекомендуються як постійна модель.
3. Доступ до внутрішнього порталу надається лише після підтвердження ролі співробітника.
4. Після зміни посади або звільнення доступ має бути змінений або відкликаний без затримки.
5. Документи з чутливим вмістом не повинні бути доступні без авторизації.

## Життєвий цикл доступу

### 1. Створення доступу

Доступ створюється, якщо:

- співробітник прийнятий на роботу;
- визначена його посада;
- визначена його роль у системі;
- відповідальний підтвердив необхідний рівень доступу.

### 2. Зміна доступу

Доступ змінюється, якщо:

- співробітник переведений на іншу посаду;
- змінена зона відповідальності;
- потрібно тимчасово надати доступ до окремого розділу.

### 3. Відкликання доступу

Доступ відкликається, якщо:

- співробітник звільнений;
- завершено тимчасовий доступ;
- виявлено порушення правил доступу;
- обліковий запис не повинен більше мати доступу до внутрішніх матеріалів.

## Хто керує доступами

| Дія | Хто відповідає |
|---|---|
| Створення акаунта | Admin / відповідальний IT або digital |
| Призначення ролі | HQ / уповноважений керівник |
| Підтвердження бізнес-потреби | Керівник функції / директор / HQ |
| Відкликання доступу при звільненні | HQ + Admin |
| Аудит доступів | HQ / Admin |

## Мінімальні вимоги безпеки

- персональні логіни для кожного користувача;
- надійні паролі;
- бажано 2FA для `admin`, `hq` і `director`;
- заборона індексації внутрішніх сторінок;
- журнал входів і критичних дій;
- закриття доступу до вкладень без авторизації;
- резервне копіювання внутрішнього контенту.

## Заборонені дії

- ❌ Використовувати один спільний пароль для всіх директорів або всієї команди.
- ❌ Змішувати публічний і внутрішній контент без розмежування доступу.
- ❌ Давати продавцю доступ до директорських або чутливих документів «про всяк випадок».
- ❌ Залишати доступ активним після звільнення або переведення співробітника.
- ❌ Публікувати внутрішні регламенти в індексованій зоні сайту.

## Етапи впровадження

### Етап 1 — Базова модель

- виділити внутрішній портал окремо від публічного сайту;
- впровадити авторизацію;
- завести базові ролі;
- закрити директорські та управлінські розділи.

### Етап 2 — Нормалізація доступів

- описати всі розділи та їхніх власників;
- накласти матрицю доступів;
- прибрати зайві ручні винятки;
- відокремити чернетки від затверджених документів.

### Етап 3 — Керована система доступів

- додати аудит доступів;
- додати журнал дій;
- додати 2FA для критичних ролей;
- впровадити регулярний перегляд ролей і доступів.

## Практична рекомендація для старту

Найкращий стартовий підхід:

1. Не вбудовувати внутрішні регламенти прямо в публічний сайт без окремого контуру доступу.
2. Використати окремий внутрішній портал або wiki.
3. Одразу виділити окремий закритий блок для директорів.
4. Зафіксувати матрицю ролей до початку технічної реалізації.
