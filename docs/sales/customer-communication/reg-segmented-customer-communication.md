# Регламент: Сегментована комунікація з клієнтами

> [!WARNING]
> Цей документ має статус `draft`. Не є офіційним до підтвердження редактором.

## Мета

Встановити єдину рамку для ручної сегментованої комунікації з клієнтами, щоб:

- команда працювала не масово, а пріоритетно;
- ресурси магазину концентрувалися на цінних клієнтах;
- кожен сценарій мав чітке джерело даних, формат контакту та статуси;
- персональні оффери та бонуси відповідали цінності клієнта;
- клієнтська комунікація була людською, простою та однаковою за стандартом.

## Область застосування

Документ застосовується до ручних сценаріїв комунікації з клієнтами для:

- привітання з днем народження;
- нагадування про згоряння бонусів;
- м'якої реактивації клієнтів, які втрачають активність.

Документ є верхньорівневим регламентом. Детальні кроки, статуси та сценарії описані в окремих профільних регламентах.

## Базові принципи

### 1. Не працюємо однаково з усіма

Ручна комунікація не запускається масово на всю базу. Пріоритет отримують клієнти, для яких очікуваний бізнес-результат перевищує витрати часу команди.

### 2. Працюємо тільки від джерела правди

Для кожного сценарію є окреме джерело даних. Команда не формує списки «з пам'яті», не копіює старі списки вручну та не використовує неактуальні вибірки.

### 3. Спочатку пріоритет, потім контакт

Перед контактом команда визначає:

- сегмент клієнта;
- силу клієнта всередині сегмента;
- очікувану вигоду від контакту;
- доречний рівень контакту;
- допустимий оффер.

### 4. Комунікація має бути людською

Клієнту не транслюються внутрішні технічні формулювання на кшталт:

- `At Risk`;
- `Sleeping`;
- «ризик втрати сегмента»;
- «низька активність по CRM».

### 5. Кожна дія має бути зафіксована

Кожен ручний контакт має завершуватися статусом. Усні домовленості без фіксації не вважаються завершеним процесом.

## Логіка відбору клієнтів

Рішення про ручне опрацювання приймається на основі комбінації факторів:

| Критерій | Що оцінюємо |
|---|---|
| Сегмент | `VIP`, `Loyal`, `At Risk`, `Sleeping`, разові |
| Середній чек | Наскільки клієнт цінний у поточній економіці магазину |
| CLV | Історична цінність клієнта |
| Давність останньої покупки | Ознака втрати активності або вікно для реактивації |
| Сума бонусів до згоряння | Чи є достатній стимул для повернення |
| Потенціал повернення | Чи є сенс у ручному контакті |

> [!IMPORTANT]
> Слабкі, одноразові та низькоцінні клієнти не повинні оброблятися так само, як `VIP` або сильні `Loyal`.

## Джерела правди за сценаріями

| Сценарій | Основне джерело | Що перевіряємо додатково |
|---|---|---|
| День народження | Дашборд дня народження | Наявність встановленої картки в Telegram-боті |
| Згоряння бонусів | Google-таблиця по бонусах, що згорають | Актуальність суми бонусів і дедлайну |
| Ризикові клієнти | Дашборд ризикових клієнтів | Актуальність пріоритету та доцільність ручного контакту |

## Рівні контакту

| Рівень контакту | Коли застосовується | Суть |
|---|---|---|
| `Дзвінок + повідомлення` | Найцінніші клієнти та пріоритетні сценарії | Максимальна персональна увага |
| `Лише повідомлення` | Середній пріоритет або обмежений ресурс команди | Точкова реактивація без повного ручного циклу |
| `Без ручної обробки` | Низька цінність або слабкий стимул | Контакт не запускається |

## Матриця сценаріїв

| Сценарій | Кого обробляємо | Основний меседж | Формат |
|---|---|---|---|
| День народження | `VIP active`, `VIP sleeping`, `Loyal`, сильні `At Risk` | Привітання, персональна увага, бонус / подарунок, запрошення прийти найближчими днями | Перевірка картки → дзвінок → ручне повідомлення |
| Згоряння бонусів | `VIP`, `Loyal`, `At Risk`, частина `Sleeping` при суттєвій сумі | Втрата вигоди, сума бонусів, дедлайн використання, запрошення прийти до дати | Для пріоритетних — дзвінок + повідомлення, для інших — повідомлення |
| Ризикові клієнти | `VIP sleeping`, `Loyal`, які втрачають активність, `At Risk`, верхній шар `Sleeping` | М'яке повернення: давно не бачили, хочемо запросити знову, є персональна увага / бонус / пропозиція | Для топ-клієнтів — дзвінок + повідомлення, для інших — ручне повідомлення |

## Правила офферів за сегментами

| Сегмент | Формат контакту | Базова логіка оффера |
|---|---|---|
| `VIP active` | Дзвінок + повідомлення | `1000 бонусів` + подарунок / комплімент |
| `VIP sleeping` | Дзвінок + повідомлення | `1000 бонусів` + подарунок або сильний реактиваційний оффер |
| `Loyal` | Дзвінок або повідомлення залежно від сили клієнта | `500–1000 бонусів` залежно від середнього чеку і цінності |
| `At Risk` | Повідомлення або дзвінок вибірково | Переважно `500 бонусів`, інколи більше для сильніших клієнтів |
| `Sleeping / слабкі / разові` | Переважно без ручної обробки або лише повідомлення за доцільністю | Мінімальний сценарій або без окремого оффера |

> [!NOTE]
> Офер не визначається «на око». Він має відповідати сегменту, цінності клієнта та сценарію контакту.

## Єдині правила для команди

| Питання | Правило |
|---|---|
| Масовий підхід | Не працюємо однаково з усіма. Пріоритет — цінні клієнти |
| Джерело списку | Для кожного сценарію використовуємо тільки його джерело правди |
| Мова комунікації | Людська, проста, без внутрішніх технічних термінів |
| Рівень контакту | Обирається за цінністю клієнта та очікуваним результатом |
| Фіксація результату | Кожен контакт завершується статусом |
| Оффер | Відповідає сегменту й не суперечить затвердженим правилам |

## Заборонені дії

- ❌ Використовувати один і той самий сценарій для всіх клієнтів без пріоритезації.
- ❌ Брати список не з визначеного джерела правди.
- ❌ Використовувати технічні CRM-формулювання у клієнтському тексті.
- ❌ Обіцяти бонус або умови, яких немає в погодженому сегментному оффері.
- ❌ Завершувати роботу без фіксації статусу.
