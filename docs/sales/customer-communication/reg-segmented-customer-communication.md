# Рамкове правило: Сегментована комунікація з клієнтами

> [!WARNING]
> Цей документ має статус `draft`. Не є офіційним до підтвердження редактором.

## Мета

Встановити єдині рамкові правила ручної комунікації з клієнтами, щоб:

- не працювати однаково з усією базою;
- концентрувати ресурс команди на клієнтах із вищою цінністю;
- відокремити загальні правила від подієвих регламентів;
- забезпечити однаковий стандарт тону, пріоритетів і прийняття рішень.

## Область застосування

Документ застосовується до всіх ручних сценаріїв комунікації з клієнтами в межах підрозділу `sales/customer-communication`.

Це рамковий документ. Він не описує покрокові сценарії подій. Детальні дії, статуси, ролі та винятки описуються в окремих регламентах комунікації по подіях.

## Що регулює цей документ

Цей документ визначає:

- принципи відбору клієнтів у ручну роботу;
- загальну логіку пріоритетів;
- допустимі рівні контакту;
- red flags для дзвінків;
- джерела правди за типами подій;
- карту подієвих регламентів.

Цей документ не дублює:

- конкретні сценарії по днях народження;
- конкретні сценарії по згорянню бонусів;
- конкретні сценарії реактивації по ризикових клієнтах;
- майбутні окремі подієві сценарії.

## Базові принципи

1. **Не працюємо однаково з усіма.** Ручна комунікація застосовується тільки там, де вона виправдана цінністю клієнта та очікуваним результатом.
2. **Працюємо тільки від джерела правди.** Для кожної події використовується своє затверджене джерело даних.
3. **Спочатку пріоритет, потім контакт.** Команда спочатку оцінює доречність контакту, а вже потім обирає канал і оффер.
4. **Комунікація має бути людською.** У клієнтському тексті не використовуються внутрішні технічні формулювання, назви сегментів і CRM-терміни.
5. **Кожна дія має бути зафіксована.** Ручний контакт не вважається завершеним без статусу або іншої форми внутрішньої фіксації.

## Критерії пріоритету

Рішення про ручне опрацювання клієнта приймається на основі сукупності факторів:

| Критерій | Що оцінюємо |
|---|---|
| Сегмент | Рівень цінності клієнта для бізнесу |
| Середній чек | Поточна комерційна вага клієнта |
| CLV | Історична цінність клієнта |
| Давність останньої покупки | Ознака активності або втрати контакту |
| Подія / привід | Чи є доречна причина для контакту |
| Потенціал повернення | Чи виправдане ручне опрацювання |

> [!IMPORTANT]
> Слабкі, низькоцінні та одноразові клієнти не повинні опрацьовуватися так само, як `VIP` або сильні `Loyal`.

## Рівні контакту

| Рівень контакту | Коли застосовується |
|---|---|
| `Дзвінок + повідомлення` | Найцінніші клієнти та найпріоритетніші події |
| `Лише повідомлення` | Середній пріоритет або обмежений ресурс команди |
| `Без ручної обробки` | Низька цінність або відсутність достатнього приводу |

## Red flags для дзвінків

| Умова | Правило |
|---|---|
| Сегмент нижче `Loyal` | Не телефонувати |
| Остання покупка була `24+` місяці тому | Не телефонувати |

## Єдині правила для команди

| Питання | Правило |
|---|---|
| Масовий підхід | Не застосовуємо один і той самий ручний сценарій до всіх клієнтів |
| Джерело списку | Для кожної події використовуємо тільки визначене джерело правди |
| Мова комунікації | Людська, проста, без внутрішніх технічних термінів |
| Рівень контакту | Визначається цінністю клієнта та силою події |
| Фіксація результату | Кожен контакт має бути зафіксований у відповідному процесі |
| Ескалація сумнівів | Якщо дані або доцільність контакту неочевидні — питання ескалується керівнику |
