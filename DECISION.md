# DECISION

## Decision: захист wiki через Authelia + Caddy

### Статус
Прийнято.

### Дата
2026-03-26

### Контекст
Потрібно було закрити корпоративну wiki від неавторизованих користувачів і дати щонайменше два рівні доступу:
- `admin`
- `user`

Також є вимога мати окрему захищену гілку/секцію wiki, недоступну звичайним користувачам.

### Розглянуті варіанти
1. `Basic Auth` у Caddy
2. `Cloudflare Access`
3. `Authentik + Caddy`
4. `Authelia + Caddy`

### Чому обрано `Authelia + Caddy`
- self-hosted варіант без прив'язки до зовнішнього SaaS
- добре підходить для внутрішньої wiki
- дозволяє централізовано закривати весь сайт перед application layer
- підтримує логін, сесії, групи, розмежування доступу
- добре масштабується на path-based політики, напр. `/admin/*`
- простіший за Authentik для цього конкретного кейсу

### Принципи реалізації
- Захист робиться **перед сайтом**, а не всередині VitePress UI.
- VitePress не розглядається як джерело прав доступу.
- Контроль доступу має бути на рівні reverse proxy / auth gateway.
- Бажаний спосіб сегментації — **path-based**, а не окремі subdomain'и для ролей.

### Базова рольова модель
- `admins`
- `users`

### Базова політика доступу
- `wiki.secretshop.ua/*` → `users`, `admins`
- `wiki.secretshop.ua/admin/*` → тільки `admins`
- неавторизованим користувачам доступ закритий

### Тимчасове технічне відхилення
На етапі впровадження wiki тимчасово працює через `VitePress dev server` на `127.0.0.1:5173`, проксований через `Caddy`.

Це допустимо для швидкого запуску, але не вважається фінальною production-архітектурою.

### Подальше цільове покращення
Перейти з `vitepress dev` на:
- `vitepress build`
- віддачу статичних файлів через `Caddy`

Це прибере dev-mode обмеження і спростить експлуатацію.

### Наслідки рішення
Позитивні:
- централізований контроль доступу
- можливість масштабувати auth для інших внутрішніх сервісів
- чистий поділ між контентом wiki і системою авторизації

Негативні / вартість:
- треба окремо підтримувати Authelia
- з'являється додаткова інфраструктурна складність
- потрібен коректний Docker/runtime setup на сервері
